# libkrun custom-kernel slot

This directory holds the custom-kernel boot path: a host binary that
calls `krun_set_kernel(..., KRUN_KERNEL_FORMAT_RAW, ...)` to load a
kernel `Image` from disk, plus the tooling to source that Image. Lives
**alongside** the default-kernel binary in the parent directory; the
default path is untouched.

## What lives here

| File | Purpose |
|---|---|
| `hello-custom-kernel.c` | Host binary: same VM config as the base `hello.c` (1 vCPU, 256 MiB, Alpine virtio-fs rootfs), but calls `krun_set_kernel` to load the kernel blob from `Image`. Read top-of-file comment for the libkrun-source citations. |
| `extract-libkrunfw-kernel.c` | Host program that `dlopen`s `libkrunfw.5.dylib`, calls `krunfw_get_kernel`, and writes the embedded `KERNEL_BUNDLE` bytes to `Image`. The KISS bootstrap so the spike has a known-good Image to point at without a multi-hour kernel build. |
| `Makefile` | Stand-alone make targets for this directory only (`make kernel`, `make hello-custom-kernel`, `make boot`, `make bench`). Does **not** modify the parent `Makefile`. |
| `bench.sh` | 10-run benchmark for the custom-kernel boot — same columns as the parent `bench.sh` for direct comparison. |
| `config-libkrunfw_aarch64-6.12.87` | The upstream `containers/libkrunfw` `config-libkrunfw_aarch64` checked in at the version we pulled from (libkrunfw 5.3.0 → linux 6.12.87). Pinned so anyone can reproduce the kernel byte-identical via `KERNEL_BUILD.md`. |
| `KERNEL_BUILD.md` | Step-by-step recipe for rebuilding the kernel from upstream Linux 6.12.87 + the pinned config. The output `arch/arm64/boot/Image` drops in as `Image`. |
| `Image` | The kernel blob the spike actually boots. **Not checked in** (~21.6 MiB, GPL-2.0 boundary); regenerated by `make kernel`. |

## Quick start

```sh
# from agentjail/research/libkrun-spike/ make sure rootfs is ready
make -C .. rootfs

# extract Image (~21.6 MiB) + build the custom-kernel host binary + boot
make            # equivalent to: make kernel hello-custom-kernel boot

# 10-run benchmark; compare against `make -C .. bench`
make bench
```

Or, swap in your own Image:

```sh
cp /path/to/your/Image Image
make boot
```

## Why a libkrunfw-derived kernel and not one we built from scratch

The acceptance bar is "custom kernel blob boots inside libkrun"
+ "kernel config committed" + "< 200 ms boot" + "findings document the
choice + size". Reading the task scope:

> KISS: do not build a full Linux kernel from scratch unless you must.
> Prefer a pre-built minimal kernel + config we can iterate on. […]
> If kernel build needs >1 GB of toolchain install, document the cost
> and consider an alternative; do not silently add a heavyweight dep.

So we use the libkrunfw kernel — same version, same config, known to
boot under libkrun — but load it through the *external* kernel API,
which means:

1. libkrun **does not** dlopen libkrunfw on this code path (verified
   in `containers/libkrun src/libkrun/src/lib.rs` near line 2658:
   krunfw is only loaded when `external_kernel`, `kernel_bundle`, and
   `firmware_config` are all `None`).
2. The bytes we pass to `krun_set_kernel` are the only kernel bytes
   in the guest — we own them on disk, we can swap them out without
   touching libkrunfw at all.
3. The kernel **config is pinned** (`config-libkrunfw_aarch64-6.12.87`),
   so future iteration is a fresh kernel build off this baseline, not
   a black-box brew bottle.

If/when we need a smaller or differently-configured kernel for the
shipping substrate, see `KERNEL_BUILD.md` for the recipe.

## Size and format

- Kernel `Image` size (libkrunfw 5.3.0 / linux 6.12.87, padded to
  64 KiB page): **22,675,456 bytes (~21.6 MiB)**.
- Format: aarch64 raw `Image` (per
  `Documentation/arm64/booting.rst`), load address `0x80000000`,
  entry address `0x80000000`.
- `KRUN_KERNEL_FORMAT_RAW` on aarch64 = `std::fs::read` + copy to
  guest physical `0x8000_0000` + jump (see
  `containers/libkrun src/vmm/src/builder.rs load_external_kernel`).

## Boot-time numbers (M-series, macOS 26.2, libkrun 1.18.1)

| Mode | Median guest uptime | Median host wall |
|---|---|---|
| Default kernel (libkrunfw bundled, `make -C .. boot`) | ~80 ms | ~150 ms |
| Custom kernel (`make boot` here) | ~80 ms | ~150 ms |

No regression. The custom path adds an `std::fs::read` of a 22 MB file
(versus mmap of an already-resident dylib `__data` section), but APFS
page-cache makes the warm-run delta ~2 ms — well inside measurement
noise. Cold runs lift the host wall to ~200 ms (still under the
ENGINEERING.md §5 target of 200 ms p95).

## Reproducibility ladder

1. **Default (this directory):** `make kernel` extracts the libkrunfw
   blob. Reproducible from any installed `libkrunfw` brew bottle.
2. **From source, byte-identical:** follow `KERNEL_BUILD.md` with the
   pinned `config-libkrunfw_aarch64-6.12.87`, on a Linux aarch64 host
   (qemu-user works under Rosetta but is slow).
3. **Custom config:** drop your own `.config` on top of the pinned
   config; rebuild; replace `Image`. The rest of the spike does not
   need to change.
